下面是一份适合你当前 **Avalonia + Dock.Avalonia 基础编辑器** 的架构方案。目标是：前期保持单项目开发效率，后期可以平滑拆成多个项目或插件模块。

# Avalonia 编辑器架构方案 v1

## 1. 架构目标

本项目定位为一个可扩展的桌面编辑器框架，核心能力包括：

```text
1. 基于 Dock.Avalonia 的高级停靠布局
2. 支持文档标签页、工具面板、Split 分割、浮动窗口
3. 支持完全自定义 Dock 与基础 UI 样式
4. 支持按 Feature 扩展功能，例如资源管理器、文本编辑器、搜索、输出、问题列表
5. 支持布局、设置、最近文件、工作区状态持久化
6. 保持 UI、业务逻辑、基础服务、具体功能之间的清晰边界
```

架构采用 **MVVM + Shell + Features + Core + Infrastructure + UI** 的分层方式。Avalonia 官方文档也建议用 MVVM 分离界面与应用逻辑，View 和 ViewModel 通过数据绑定通信；Avalonia 模板中的 `ViewLocator` 则是常见的 ViewModel-first 显示方式。([Avalonia 文档][1])

## 3. 分层职责

### 3.1 Core：核心抽象层

`Core` 是整个项目最底层的公共协议层。它不应该引用 Avalonia 控件，也不应该依赖具体文件系统或 Dock 控件实现。

```text
│   ├── IPanelRegistry.cs
│   ├── ICommandService.cs
│   ├── IEventBus.cs
│   ├── IFileSystemService.cs
│   ├── ISettingsService.cs
│   ├── IDocumentService.cs
│   └── IWorkspaceService.cs
│
├── Models
│   ├── FilePath.cs
│   ├── Workspace.cs
│   ├── EditorDocument.cs

│   ├── TextRange.cs
```

Avalonia 桌面应用的主窗口通常通过 `ApplicationLifetime.MainWindow` 设置；因此这里的 `MainWindow` 就是应用 Shell，也就是整个编辑器的根窗口。([Avalonia 文档][3])

`Shell` 负责：

```text
1. 创建主窗口
2. 组合 Dock 布局
3. 加载 Feature 提供的面板
4. 管理菜单、工具栏、状态栏
5. 注册全局快捷键
6. 路由全局命令
7. 保存和恢复 Dock 布局
8. 协调文档区和工具区
```

`Shell` 不负责：

```text
1. 文件树怎么加载
2. 文本编辑器怎么编辑
3. 搜索怎么执行
4. 输出日志怎么产生
5. 问题列表怎么解析
6. 设置项具体怎么保存
```

这些都应该放到各自的 `Feature` 或 `Infrastructure`。

## 4. 依赖方向

推荐依赖方向：

```text
App
 ├── Shell
 ├── Features
 ├── Infrastructure
 ├── UI
 └── Core

Shell
 ├── Core
 ├── UI
 └── Features

Features
 ├── Core
 └── UI

Infrastructure
 └── Core

UI
 └── Core，尽量不依赖 Features

Core
 └── 不依赖其他业务层
```

禁止方向：

```text
Core              不能依赖 Shell / UI / Features / Infrastructure
Infrastructure    不建议依赖 Shell / Features
UI                不建议依赖具体 Feature
Feature A         不直接依赖 Feature B
ViewModel         不直接 new 具体 View
Shell             不写具体 Feature 的业务逻辑
```

Feature 之间需要通信时，通过 `Core` 中的抽象：

```text
IEventBus
ICommandService
IDocumentService
IWorkspaceService
IPanelRegistry
```

## 6. View / ViewModel 方案

采用 ViewModel-first。

目录命名保持：

```text
Features/TextEditor/ViewModels/TextEditorViewModel.cs
Features/TextEditor/Views/TextEditorView.axaml

Features/WorkspaceExplorer/ViewModels/WorkspaceExplorerViewModel.cs
Features/WorkspaceExplorer/Views/WorkspaceExplorerView.axaml

Shell/ViewModels/MainWindowViewModel.cs
Shell/Views/MainWindow.axaml
```

Avalonia 的 `ViewLocator` 是默认模板中用于 MVVM 的便利机制，也可以替换成显式 `DataTemplate`；这里建议前期用显式 `ViewLocator`，避免反射字符串查找在重构目录时出错。([Avalonia 文档][6])

示例：

```csharp
using Avalonia.Controls;
using Avalonia.Controls.Templates;
using Editor.Core.ViewModels;
using Editor.Features.TextEditor.ViewModels;
using Editor.Features.TextEditor.Views;
using Editor.Features.WorkspaceExplorer.ViewModels;
using Editor.Features.WorkspaceExplorer.Views;
using Editor.Shell.ViewModels;
using Editor.Shell.Views;

namespace Editor;

public sealed class ViewLocator : IDataTemplate
{
    public Control Build(object? data)
    {
        return data switch
        {
            MainWindowViewModel => new MainWindow(),
            TextEditorViewModel => new TextEditorView(),
            WorkspaceExplorerViewModel => new WorkspaceExplorerView(),
            _ => new TextBlock
            {
                Text = data is null
                    ? "ViewModel is null"
                    : $"View not found: {data.GetType().FullName}"
            }
        };
    }

    public bool Match(object? data)
    {
        return data is ViewModelBase;
    }
}
```

`App.axaml`：

```xml
<Application xmlns="https://github.com/avaloniaui"
             xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
             xmlns:local="using:Editor"
             xmlns:ui="using:Editor.UI.Styles"
             x:Class="Editor.App"
             RequestedThemeVariant="Dark">

  <Application.DataTemplates>
    <local:ViewLocator />
  </Application.DataTemplates>

  <Application.Styles>
```

## 7. DI 与启动流程

建议引入 `.NET Generic Host` 或至少使用 `Microsoft.Extensions.DependencyInjection`。

.NET 的 DI 通过接口或基类抽象依赖，并在启动时注册到 `IServiceCollection`，构建后通过 `IServiceProvider` 解析服务；这样可以减少硬编码依赖，提高可测试性。([Microsoft Learn][7])

推荐包：

```bash
dotnet add package Microsoft.Extensions.DependencyInjection
dotnet add package Microsoft.Extensions.Hosting
dotnet add package Microsoft.Extensions.Logging
dotnet add package CommunityToolkit.Mvvm
```

`CommunityToolkit.Mvvm` 可以用于 `ObservableObject`、`RelayCommand`、`AsyncRelayCommand`。官方文档说明 `RelayCommand` / `RelayCommand<T>` 是 `ICommand` 实现，用于把 ViewModel 中的方法暴露给 UI 绑定。([Microsoft Learn][8])

### 7.1 AppBootstrapper

```text
AppBootstrapper.cs
```

```csharp
using Editor.Core.Abstractions;
using Editor.Features.WorkspaceExplorer;
using Editor.Infrastructure.FileSystem;
using Editor.Infrastructure.Settings;
using Editor.Shell.Docking;
using Editor.Shell.ViewModels;
using Microsoft.Extensions.DependencyInjection;

namespace Editor;

public static class AppBootstrapper
{
    public static IServiceProvider ConfigureServices()
    {
        var services = new ServiceCollection();

        RegisterCore(services);
        RegisterInfrastructure(services);
        RegisterShell(services);
        RegisterFeatures(services);

        return services.BuildServiceProvider();
    }

    private static void RegisterCore(IServiceCollection services)
    {
        services.AddSingleton<IPanelRegistry, PanelRegistry>();
        services.AddSingleton<ICommandService, CommandService>();
        services.AddSingleton<IEventBus, EventBus>();
    }

    private static void RegisterInfrastructure(IServiceCollection services)
    {
        services.AddSingleton<IFileSystemService, LocalFileSystemService>();
        services.AddSingleton<ISettingsService, JsonSettingsService>();
    }

    private static void RegisterShell(IServiceCollection services)
    {
        services.AddSingleton<EditorDockFactory>();
        services.AddSingleton<DockLayoutService>();
        services.AddSingleton<DockLayoutPersistenceService>();

        services.AddTransient<MainWindowViewModel>();
    }

    private static void RegisterFeatures(IServiceCollection services)
    {
        services.AddSingleton<IEditorModule, WorkspaceExplorerModule>();
        services.AddSingleton<IEditorModule, TextEditorModule>();
        services.AddSingleton<IEditorModule, SearchModule>();
        services.AddSingleton<IEditorModule, OutputModule>();
        services.AddSingleton<IEditorModule, ProblemsModule>();
    }
}
```

### 7.2 App.axaml.cs

```csharp
using Avalonia;
using Avalonia.Controls.ApplicationLifetimes;
using Avalonia.Markup.Xaml;
using Editor.Core.Abstractions;
using Editor.Shell.ViewModels;
using Editor.Shell.Views;
using Microsoft.Extensions.DependencyInjection;

namespace Editor;

public partial class App : Application
{
    public static IServiceProvider Services { get; private set; } = null!;

    public override void Initialize()
    {
        AvaloniaXamlLoader.Load(this);
    }

    public override void OnFrameworkInitializationCompleted()
    {
        Services = AppBootstrapper.ConfigureServices();

        RegisterModules(Services);

        if (ApplicationLifetime is IClassicDesktopStyleApplicationLifetime desktop)
        {
            desktop.MainWindow = new MainWindow
            {
                DataContext = Services.GetRequiredService<MainWindowViewModel>()
            };
        }

        base.OnFrameworkInitializationCompleted();
    }

    private static void RegisterModules(IServiceProvider services)
    {
        var panels = services.GetRequiredService<IPanelRegistry>();
        var commands = services.GetRequiredService<ICommandService>();

        foreach (var module in services.GetServices<IEditorModule>())
        {
            module.RegisterPanels(panels);
            module.RegisterCommands(commands);
        }
    }
}
```

## 9. Dock 布局服务设计

### 9.1 EditorDockFactory 职责

```text
EditorDockFactory
├── 从 IPanelRegistry 读取所有注册面板
├── 按 DockArea 分配到 Left / Right / Bottom / Center
├── 创建 ToolDock / DocumentDock
├── 创建 ProportionalDock 和 Splitter
├── 设置 CanClose / CanFloat / CanDrag / CanDrop
└── 提供打开文档、关闭面板、移动面板等能力
```

伪代码：

```csharp
public sealed class EditorDockFactory : Factory
{
    private readonly IPanelRegistry _panels;
    private readonly IServiceProvider _services;

    public EditorDockFactory(
        IPanelRegistry panels,
        IServiceProvider services)
    {
        _panels = panels;
        _services = services;
    }

    public override IRootDock CreateLayout()
    {
        var leftPanels = _panels.GetAll()
            .Where(x => x.DefaultArea == DockArea.Left)
            .ToList();

        var bottomPanels = _panels.GetAll()
            .Where(x => x.DefaultArea == DockArea.Bottom)
            .ToList();

        var rightPanels = _panels.GetAll()
            .Where(x => x.DefaultArea == DockArea.Right)
            .ToList();

        var documentPanels = _panels.GetAll()
            .Where(x => x.DefaultArea == DockArea.Center)
            .ToList();

        // 在这里创建 ToolDock / DocumentDock / ProportionalDock
        // 保持 DockFactory 只做布局编排，不写 Feature 业务逻辑

        return rootDock;
    }
}
```

### 9.2 DockLayoutPersistenceService

职责：

```text
1. 启动时恢复上次 Dock 布局
2. 没有历史布局时使用默认布局
3. 退出时保存 Dock 布局
4. 版本变更时处理布局迁移
```

建议保存路径：

```text
%AppData%/Editor/layout/dock-layout.json
%AppData%/Editor/settings/settings.json
%AppData%/Editor/workspaces/recent.json
```

Dock.Avalonia 支持布局保存与恢复，也提供多种序列化包选择；如果你用 JSON，建议封装到自己的 `DockLayoutPersistenceService`，不要让业务代码直接依赖具体序列化库。([Wieslaw Soltes][4])

# 18. 编辑器外壳性能指南

## 18.1 外壳层性能目标

编辑器外壳的目标不是处理引擎渲染，而是保证 UI 框架本身在大量窗口、面板、文档、命令、属性项、日志项、资源项存在时仍然流畅。

核心原则：

```text
1. Shell 只负责组织，不承载重业务
2. Dock 面板按需创建
3. 文档内容按需加载
4. 大列表必须虚拟化
5. 绑定必须可编译、可检查
6. UI 线程只做 UI 更新
7. 高频 UI 更新必须批处理
8. 面板关闭必须释放资源
9. 性能问题必须可观测
```

Avalonia 官方性能文档明确建议在大数据集合场景使用虚拟化、优化视觉树结构、减少绑定错误、异步加载数据，并指出虚拟化依赖控件所在容器能提供有限尺寸；例如把列表放进 `StackPanel` 可能会导致虚拟化失效。([Avalonia 文档][1])

## 18.3 Shell 启动路径要最小化

启动阶段只做这些：

```text
1. 创建 App
2. 创建 MainWindow
3. 加载基础主题
4. 创建 Dock 根布局
5. 注册面板描述
6. 创建默认可见的轻量面板
7. 恢复布局 ID
```

启动阶段不要做这些：

```text
1. 创建所有 Dock 面板内容
2. 创建所有 Feature 的 ViewModel
3. 创建所有文档 View
4. 加载所有菜单子项
5. 初始化所有工具窗口
6. 扫描所有工作区数据
7. 批量构造大量 UI item
```

Dock.Avalonia 本身提供 document panes、tool panes、floating windows、docking targets 和 layout persistence，所以 Shell 可以只保存“布局结构 + 面板 ID”，不需要在启动时提前创建所有面板内容。([Wieslaw Soltes][2])

## 18.4 Dock 面板采用描述符优先

不要让 `DockFactory` 直接 new 所有面板的 ViewModel。

不推荐：

```csharp
var explorer = new WorkspaceExplorerViewModel();
var inspector = new InspectorViewModel();
var output = new OutputViewModel();
var profiler = new ProfilerViewModel();
```

推荐用轻量描述符：

```csharp
public sealed record PanelDescriptor(
    string Id,
    string Title,
    PanelKind Kind,
```

## 18.5 LazyDockableContext

给 Dockable 的 `Context` 包一层懒加载对象。

```csharp
public sealed class LazyDockableContext
{
    private readonly IServiceProvider _services;
    private readonly Func<IServiceProvider, object> _factory;
    private object? _instance;

    public LazyDockableContext(
        IServiceProvider services,
        Func<IServiceProvider, object> factory)
    {
        _services = services;
        _factory = factory;
    }

    public object Value => _instance ??= _factory(_services);

    public bool IsCreated => _instance is not null;

    public bool TryDispose()
    {
        if (_instance is IDisposable disposable)
        {
            disposable.Dispose();
            _instance = null;
            return true;
        }

        _instance = null;
        return false;
    }
}
```

Dockable 创建时：

```csharp
var context = new LazyDockableContext(
    services,
    descriptor.CreateViewModel);

var tool = new Tool
{
    Id = descriptor.Id,
    Title = descriptor.Title,
    Context = context,
    CanClose = true,
    CanFloat = true,
    CanDrag = true,
    CanDrop = true
};
```

然后通过 DataTemplate 显示真正内容：

```xml
<DataTemplate DataType="shellDocking:LazyDockableContext">
  <ContentControl Content="{Binding Value}" />
</DataTemplate>
```

这样 Shell 可以先创建 Dock 面板壳，真正的 ViewModel 和 View 到第一次显示时才创建。

## 18.9 Shell 集合更新要批量化

外壳层常见问题是频繁刷新集合：

```text
状态栏每帧刷新

Problems 每个诊断 Add 一次

Recent Files 每次操作刷新整个菜单
Open Documents 每个文档状态变化都触发完整刷新
```

建议使用批量集合：

```csharp
using System.Collections.ObjectModel;
using System.Collections.Specialized;

public sealed class RangeObservableCollection<T> : ObservableCollection<T>
{
    private bool _suppressNotification;

    public void ReplaceRange(IEnumerable<T> items)
    {
        try
        {
            _suppressNotification = true;

            ClearItems();

            foreach (var item in items)
            {
                Items.Add(item);
            }
        }
        finally
        {
            _suppressNotification = false;
        }

        OnPropertyChanged(new(nameof(Count)));
        OnPropertyChanged(new("Item[]"));
        OnCollectionChanged(
            new NotifyCollectionChangedEventArgs(
                NotifyCollectionChangedAction.Reset));
    }

    public void AddRange(IEnumerable<T> items)
    {
        var added = items.ToList();

        if (added.Count == 0)
        {
            return;
        }

        var startIndex = Count;

        try
        {
            _suppressNotification = true;

            foreach (var item in added)
            {
                Items.Add(item);
            }
        }
        finally
        {
            _suppressNotification = false;
        }

        OnPropertyChanged(new(nameof(Count)));
        OnPropertyChanged(new("Item[]"));
        OnCollectionChanged(
            new NotifyCollectionChangedEventArgs(
                NotifyCollectionChangedAction.Add,
                added,
                startIndex));
    }

    protected override void OnCollectionChanged(NotifyCollectionChangedEventArgs e)
    {
        if (!_suppressNotification)
        {
            base.OnCollectionChanged(e);
        }
    }
}
```

使用规则：

```text
少量变化：普通 ObservableCollection
大量替换：ReplaceRange
大量追加：AddRange
持续流式输入：先进入 buffer，再定时 flush
超大集合：分页 / 虚拟数据源
```

## 18.10 开启编译绑定

外壳层控件多、模板多、DataTemplate 多，绑定错误和反射绑定都会扩大成本。

项目级开启：

```xml
<PropertyGroup>
  <AvaloniaUseCompiledBindingsByDefault>true</AvaloniaUseCompiledBindingsByDefault>
</PropertyGroup>
```

Avalonia 文档说明，可以通过 `AvaloniaUseCompiledBindingsByDefault` 让编译绑定成为默认行为；启用后绑定需要声明 `x:DataType`，需要动态解析的个别绑定可以显式使用 `ReflectionBinding`。([Avalonia 文档][4])

View 写法：

```xml
<UserControl xmlns="https://github.com/avaloniaui"
             xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
             xmlns:vm="using:Editor.Shell.ViewModels"
             x:Class="Editor.Shell.Views.MainShellView"
             x:DataType="vm:MainShellViewModel">

  <TextBlock Text="{Binding Title}" />

</UserControl>
```

DataTemplate 也要写：

```xml
<DataTemplate x:DataType="commands:CommandItemViewModel">
  <TextBlock Text="{Binding Title}" />
</DataTemplate>
```

`x:DataType` 是编译绑定的类型基础，Avalonia 文档明确说明它用于指定绑定作用域内的预期数据类型，并且是 compiled binding 所需的信息。([Avalonia 文档][5])

## 18.11 Binding Error 零容忍

外壳层必须把 binding error 当成性能问题处理，而不是只当成警告。

Avalonia 性能文档明确提到，绑定错误会影响性能，因为每次绑定失败都会尝试解析并写入 trace log。([Avalonia 文档][1])

规范：

```text
1. Debug 启动后不允许有 binding error
2. Shell View 必须写 x:DataType
3. DataTemplate 必须写 x:DataType
4. 大量 item template 禁止 ReflectionBinding
5. ReflectionBinding 只能用于确实动态的属性面板
6. 避免在 item template 中大量使用 RelativeSource.FindAncestor
7. 不在 converter 里访问服务或做 IO
```

Avalonia 调试文档也建议使用 DevTools、Compiled Bindings、LogToTrace 等工具来检查视觉树、样式、事件和绑定问题。([Avalonia 文档][6])

## 18.15 UI 线程只处理最终 UI 更新

Avalonia 使用 UI 线程模型；官方文档说明，如果需要从后台线程更新 UI，应通过 `Dispatcher.UIThread.Post` 或 `InvokeAsync` 把工作切回 UI 线程。([Avalonia 文档][7])

不要这样：

```csharp
foreach (var item in items)
{
    Dispatcher.UIThread.Post(() =>
    {
        Items.Add(item);
    });
}
```

这会把大量小任务塞进 UI 队列。

推荐用 Shell 级 UI 批处理器：

```csharp
using System.Collections.Concurrent;
using Avalonia.Threading;

public sealed class UiBatcher
{
    private readonly ConcurrentQueue<Action> _queue = new();
    private int _posted;

    public void Enqueue(Action action)
    {
        _queue.Enqueue(action);

        if (Interlocked.Exchange(ref _posted, 1) == 0)
        {
            Dispatcher.UIThread.Post(Flush, DispatcherPriority.Background);
        }
    }

    private void Flush()
    {
        try
        {
            while (_queue.TryDequeue(out var action))
            {
                action();
            }
        }
        finally
        {
            Interlocked.Exchange(ref _posted, 0);

            if (!_queue.IsEmpty &&
                Interlocked.Exchange(ref _posted, 1) == 0)
            {
                Dispatcher.UIThread.Post(Flush, DispatcherPriority.Background);
            }
        }
    }
}
```

使用方式：

```csharp
var result = await Task.Run(() =>
{
    return BuildCommandSearchResult(keyword);
});

_uiBatcher.Enqueue(() =>
{
    SearchResults.ReplaceRange(result);
});
```

外壳层准则：

```text
后台线程：
  搜索
  排序
  过滤
  配置读取
  菜单数据准备
  日志聚合

UI 线程：
  ReplaceRange
  设置绑定属性
  打开 Dock 面板
  激活文档
  显示弹窗
```

## 18.17 事件订阅集中管理

## 18.19 Command Palette 性能策略

命令面板是 Shell 级功能，后续命令会越来越多。

推荐结构：

```text
Features/CommandPalette
├── ViewModels
│   ├── CommandPaletteViewModel.cs
│   └── CommandSearchItemViewModel.cs
├── Services
│   └── CommandSearchService.cs
└── Views
    └── CommandPaletteView.axaml
```

策略：

```text
1. 命令注册表常驻
2. 搜索结果临时生成
3. 搜索输入 debounce
4. 搜索结果最多先显示前 N 条
5. UI 结果列表虚拟化
6. 不为每个命令常驻一个复杂 ViewModel
```

示例：

```csharp
public sealed class CommandPaletteViewModel : ViewModelBase
{
    private readonly ICommandSearchService _search;
    private CancellationTokenSource? _searchCts;

    public RangeObservableCollection<CommandSearchItemViewModel> Results { get; } = [];

    public async Task SearchAsync(string keyword)
    {
        _searchCts?.Cancel();
        _searchCts = new CancellationTokenSource();

        var token = _searchCts.Token;

        await Task.Delay(120, token);

        var results = await Task.Run(
            () => _search.Search(keyword, limit: 100),
            token);

        Results.ReplaceRange(
            results.Select(x => new CommandSearchItemViewModel(x)));
    }
}
```

## 18.20 Output / Problems 这类 Shell 面板要设上限

即使它们属于 Feature，也通常挂在 Shell 的底部 Dock 区域，需要外壳级性能约束。

Output：

```text
1. 日志先进入后台 buffer
2. 每 50ms / 100ms 批量 flush
3. UI 只显示最近 N 条
4. 完整日志写文件
5. 清空日志时释放旧对象
```

Problems：

```text
1. 全量诊断结果一次 ReplaceRange
2. 分组和排序放后台
3. 过滤结果分页或虚拟化
4. 不给每条 problem 创建多个命令对象
```

示例：

```csharp
public sealed class OutputPanelViewModel : ShellPanelViewModelBase
{
    private const int MaxVisibleMessages = 5000;

    private readonly List<OutputMessageViewModel> _pending = [];

    public RangeObservableCollection<OutputMessageViewModel> Messages { get; } = [];

    public override string Id => "output";

    public void Append(OutputMessage message)
    {
        lock (_pending)
        {
            _pending.Add(new OutputMessageViewModel(message));
        }
    }

    public void Flush()
    {
        List<OutputMessageViewModel> batch;

        lock (_pending)
        {
            if (_pending.Count == 0)
            {
                return;
            }

            batch = [.. _pending];
            _pending.Clear();
        }

        Messages.AddRange(batch);

        while (Messages.Count > MaxVisibleMessages)
        {
            Messages.RemoveAt(0);
        }
    }
}
```

## 18.21 内存分配规范只针对 Shell/UI

这里不讨论引擎数据，只关注 Shell/UI 对象：

## 18.25 一句话总原则

外壳层的性能边界可以总结成：

```text
Dock 保存布局，不保存重对象。
Panel 保存描述，不急着创建内容。
Document 保存状态，不强制常驻 View。
Command 保存描述，不到处 new ICommand。
List 显示可见项，不创建全部控件。
UI 线程做最终更新，不做重计算。
面板关闭必须释放，不靠 GC 碰运气。
```

也就是：

> **编辑器外壳要像一个调度器，而不是一个对象容器。**

这样底层引擎对象再多，也不会直接把 Avalonia 的控件树、绑定系统、Dock 面板和 ViewModel 数量一起拖垮。

# 18. 性能设计指南

## 18.1 核心原则

编辑器性能按优先级遵守这几条：

```text
1. UI 只展示当前可见、当前展开、当前选中、当前激活的数据
2. 引擎数据和 UI 数据分离
3. 大集合必须虚拟化
4. 大量变更必须批处理
5. 后台任务不能阻塞 UI 线程
6. 高频数据不能直接绑定到 UI
7. 所有性能优化必须能被监控和验证
```

Avalonia 官方性能文档明确建议在大数据集合场景使用虚拟化，并说明虚拟化的核心是只创建和渲染可见项；文档还特别提醒，如果列表控件被放进 `StackPanel` 等给无限高度的容器里，虚拟化会失效。([Avalonia 文档][1])

## 18.3 数据层和 UI 层必须分离

不要这样设计：

```text
Engine Entity
    = EntityViewModel
    = TreeViewItem
    = Inspector Row
    = 绑定对象
```

这会导致引擎对象越多，UI 对象也越多。

推荐这样设计：

```text
Engine World
├── EntityId
├── Component 数据
├── Transform 层级
└── Runtime 状态

SceneHierarchyIndex
├── 轻量树索引
├── EntityId
├── ParentId
├── Name
├── Depth
├── IsExpanded
└── IsVisible

SceneHierarchyViewModel
└── VisibleRows: RangeObservableCollection<EntityRowViewModel>
```

也就是说：

```text
引擎层保存完整数据
索引层保存查询结构
ViewModel 层只保存当前 UI 需要显示的数据
View 层只创建当前可见控件
```

对于场景层级树，推荐只为这些对象创建 UI VM：

```text
1. 当前展开路径上的节点
2. 当前滚动区域可见节点
3. 当前选中节点
4. 搜索结果节点
```

不要为整个场景一次性创建全部 `EntityNodeViewModel`。

## 18.4 大集合控件必须虚拟化

游戏引擎编辑器中最容易炸性能的地方：

```text
Scene Hierarchy
Asset Browser
Inspector Property Grid
Console / Output
Problems / Diagnostics
Search Results
Animation Keyframe List
Profiler Events
```

这些地方都不能用普通“无限创建控件”的方式。

Avalonia 官方性能文档说明，`ListBox`、`TreeView`、`DataGrid`、`ItemsRepeater` 支持虚拟化；虚拟化会复用滚出屏幕的控件，用它们显示新进入可视区域的项。([Avalonia 文档][1])

不要这样：

```xml
<StackPanel>
  <ListBox ItemsSource="{Binding Entities}" />
</StackPanel>
```

推荐这样：

```xml
<Grid RowDefinitions="*">
  <ListBox Grid.Row="0"
           ItemsSource="{Binding VisibleEntities}" />
</Grid>
```

原因是 `StackPanel` 会给子元素无限高度，Avalonia 官方文档明确提醒这种布局会导致虚拟化失效。([Avalonia 文档][1])

## 18.5 大量数据不要逐条 Add 到 ObservableCollection

`ObservableCollection<T>` 的优势是集合变化后可以通知 UI，Avalonia 文档也说明它实现了 `INotifyCollectionChanged`，控件可以监听添加、删除和修改通知来自动刷新。([Avalonia 文档][2])
但在游戏编辑器里，如果一次导入几万资源、展开几万 Entity、输出几万日志，不应该逐条 `Add` 并让 UI 逐条刷新。

建议实现一个批量集合：

```csharp
using System.Collections.ObjectModel;
using System.Collections.Specialized;

namespace Editor.UI.Virtualization;

public sealed class RangeObservableCollection<T> : ObservableCollection<T>
{
    private bool _suppressNotification;

    public void ReplaceRange(IEnumerable<T> items)
    {
        try
        {
            _suppressNotification = true;

            ClearItems();

            foreach (var item in items)
            {
                Items.Add(item);
            }
        }
        finally
        {
            _suppressNotification = false;
        }

        OnPropertyChanged(new(nameof(Count)));
        OnPropertyChanged(new("Item[]"));
        OnCollectionChanged(
            new NotifyCollectionChangedEventArgs(
                NotifyCollectionChangedAction.Reset));
    }

    public void AddRange(IEnumerable<T> items)
    {
        var startIndex = Count;
        var added = items.ToList();

        if (added.Count == 0)
        {
            return;
        }

        try
        {
            _suppressNotification = true;

            foreach (var item in added)
            {
                Items.Add(item);
            }
        }
        finally
        {
            _suppressNotification = false;
        }

        OnPropertyChanged(new(nameof(Count)));
        OnPropertyChanged(new("Item[]"));
        OnCollectionChanged(
            new NotifyCollectionChangedEventArgs(
                NotifyCollectionChangedAction.Add,
                added,
                startIndex));
    }

    protected override void OnCollectionChanged(NotifyCollectionChangedEventArgs e)
    {
        if (!_suppressNotification)
        {
            base.OnCollectionChanged(e);
        }
    }
}
```

使用方式：

```csharp
public RangeObservableCollection<EntityRowViewModel> VisibleEntities { get; } = [];

public void RefreshVisibleRows(IReadOnlyList<EntityRowViewModel> rows)
{
    VisibleEntities.ReplaceRange(rows);
}
```

原则：

```text
少量实时变化：ObservableCollection 可以
大量整体刷新：ReplaceRange
大量增量追加：AddRange
极大数据源：分页 / 虚拟数据源
```

## 18.6 使用编译绑定

建议项目级开启 Avalonia 编译绑定：

```xml
<PropertyGroup>
  <AvaloniaUseCompiledBindingsByDefault>true</AvaloniaUseCompiledBindingsByDefault>
</PropertyGroup>
```

然后每个 View 都写 `x:DataType`：

```xml
<UserControl xmlns="https://github.com/avaloniaui"
             xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
             xmlns:vm="using:Editor.Features.SceneHierarchy.ViewModels"
             x:Class="Editor.Features.SceneHierarchy.Views.SceneHierarchyView"
             x:DataType="vm:SceneHierarchyViewModel"
             x:CompileBindings="True">

  <ListBox ItemsSource="{Binding VisibleEntities}" />

</UserControl>
```

Avalonia 官方文档说明，`x:CompileBindings` 会启用编译绑定，绑定路径会在编译期验证，并提供更好的运行时性能；编译绑定需要 `x:DataType` 提供绑定起始类型。([Avalonia 文档][3])

`DataTemplate` 也要写类型：

```xml
<DataTemplate x:DataType="vm:EntityRowViewModel">
  <TextBlock Text="{Binding Name}" />
</DataTemplate>
```

不要在大量 item template 里依赖反射式普通绑定。

## 18.7 减少 Visual Tree 深度

游戏编辑器的每一行 UI 都可能重复几千次，所以 item template 必须非常轻。

不要这样：

```xml
<Border>
  <Grid>
    <StackPanel>
      <Border>
        <StackPanel>
          <TextBlock />
        </StackPanel>
      </Border>
    </StackPanel>
  </Grid>
</Border>
```

推荐这样：

```xml
<Grid ColumnDefinitions="Auto,*"
      Height="24">
  <PathIcon Grid.Column="0"
            Width="14"
            Height="14"
            Data="{Binding Icon}" />

  <TextBlock Grid.Column="1"
             Text="{Binding Name}"
             VerticalAlignment="Center" />
</Grid>
```

Avalonia 性能文档建议保持 XAML 视觉树简单、扁平，因为每个元素都会参与 measure / arrange 布局过程；文档还建议减少 `TextBlock` 中 `Run` 的使用、优先使用 `StreamGeometry` 而不是 `PathGeometry`、缩略图使用较小尺寸图片。([Avalonia 文档][4])

## 18.8 Inspector 性能策略

`Inspector` 是游戏编辑器里最容易慢的功能之一，因为它会根据选中对象动态生成属性 UI。

不要为所有 Entity 预生成 Inspector。

正确策略：

```text
1. 只为当前选中对象生成 Inspector
2. 多选时只生成共同属性
3. 组件折叠时不创建内部属性编辑器
4. 属性编辑器使用懒加载
5. 高频属性，例如 Transform，做节流刷新
6. 编辑中的 TextBox 不要被后台刷新打断
```

推荐结构：

```text
Features/Inspector
├── ViewModels
│   ├── InspectorViewModel.cs
│   ├── ComponentSectionViewModel.cs
│   └── PropertyRowViewModel.cs
├── PropertyAdapters
│   ├── TransformPropertyAdapter.cs
│   ├── MaterialPropertyAdapter.cs
│   └── ScriptPropertyAdapter.cs
├── PropertyEditors
│   ├── FloatEditor.axaml
│   ├── Vector3Editor.axaml
│   ├── ColorEditor.axaml
│   └── AssetReferenceEditor.axaml
└── Virtualization
    └── InspectorPropertySource.cs
```

属性刷新建议分两类：

```text
用户正在编辑的属性：由 UI 主导，提交后写回引擎
引擎实时变化的属性：节流后刷新 UI
```

例如 Transform 每帧都在变，不要每帧通知 UI：

```csharp
public sealed class InspectorRefreshScheduler
{
    private readonly TimeSpan _interval = TimeSpan.FromMilliseconds(100);
    private DateTime _lastRefresh = DateTime.MinValue;

    public bool ShouldRefresh()
    {
        var now = DateTime.UtcNow;

        if (now - _lastRefresh < _interval)
        {
            return false;
        }

        _lastRefresh = now;
        return true;
    }
}
```

## 18.9 Scene Hierarchy 性能策略

场景层级树不要直接绑定完整递归树。

推荐使用“展开状态 + 扁平可见列表”：

```text
完整场景树：
EntityId -> Children

UI 可见列表：
[
  Entity A, depth 0
  Entity A.1, depth 1
  Entity A.2, depth 1
  Entity B, depth 0
]
```

这样虚拟化控件只处理一个扁平列表，展开/折叠时重新计算可见区：

```csharp
public sealed record EntityRow(
    int EntityId,
    int Depth,
    string Name,
    bool HasChildren,
    bool IsExpanded);
```

ViewModel：

```csharp
public sealed class SceneHierarchyViewModel : ViewModelBase
{
    private readonly SceneHierarchyIndex _index;

    public RangeObservableCollection<EntityRowViewModel> VisibleRows { get; } = [];

    public void ToggleExpanded(int entityId)
    {
        _index.ToggleExpanded(entityId);

        var visibleRows = _index.BuildVisibleRows()
            .Select(row => new EntityRowViewModel(row));

        VisibleRows.ReplaceRange(visibleRows);
    }
}
```

重点：

```text
不要让每个 Entity 都常驻一个复杂 ViewModel
不要让 TreeView 自动递归创建全部节点
不要在每个节点上订阅大量事件
不要在每个节点上保存大量 Command 对象
```

## 18.10 Asset Browser 性能策略

资源浏览器通常包含：

```text
贴图
材质
模型
Prefab
动画
音频
脚本
场景
```

性能重点：

```text
1. 文件扫描放后台线程
2. 缩略图异步生成
3. 缩略图必须缓存
4. 缩略图按可见区域优先加载
5. 不显示原图，显示降采样后的 thumbnail
6. 滚动时取消不可见项的缩略图任务
```

Avalonia 性能文档说明，如果小图或缩略图直接加载原始大图再缩放，会带来性能瓶颈，因为 Avalonia 默认会按原始完整尺寸加载和解码图片。([Avalonia 文档][4])

建议：

```text
Assets/Cache/Thumbnails
├── texture_xxx_128.png
├── model_xxx_128.png
└── material_xxx_128.png
```

缩略图服务：

```csharp
public interface IThumbnailService
{
    ValueTask<Bitmap?> GetThumbnailAsync(
        string assetPath,
        int size,
        CancellationToken cancellationToken);
}
```

AssetBrowser 每个 item 不直接加载图片，而是请求缩略图服务：

```csharp
public sealed class AssetItemViewModel : ViewModelBase, IDisposable
{
    private readonly IThumbnailService _thumbnailService;
    private readonly CancellationTokenSource _cts = new();

    public Bitmap? Thumbnail { get; private set; }

    public async Task LoadThumbnailAsync()
    {
        Thumbnail = await _thumbnailService.GetThumbnailAsync(
            AssetPath,
            128,
            _cts.Token);

        OnPropertyChanged(nameof(Thumbnail));
    }

    public void Dispose()
    {
        _cts.Cancel();
        _cts.Dispose();
        Thumbnail?.Dispose();
    }
}
```

## 18.11 Dock 面板性能策略

Dock 面板不要一次性全部创建重内容。

不要这样：

```text
ShaderGraphView
所有 ViewModel
```

推荐：

```text
启动时只创建 Dock 描述
面板首次显示时创建 ViewModel
面板隐藏后保留轻状态
长期不用的重面板可以释放内容
```

面板注册信息应该是轻量的：

```csharp
public sealed record PanelDescriptor(
    string Id,
    string Title,
    PanelKind Kind,
    DockArea DefaultArea,
    Func<IServiceProvider, object> CreateViewModel,
    bool LazyLoad = true,
    bool KeepAlive = true);
```

Dockable 创建时：

```csharp
public sealed class LazyDockableContext
{
    private readonly IServiceProvider _services;
    private readonly Func<IServiceProvider, object> _factory;
    private object? _instance;

    public LazyDockableContext(
        IServiceProvider services,
        Func<IServiceProvider, object> factory)
    {
        _services = services;
        _factory = factory;
    }

    public object Value => _instance ??= _factory(_services);
}
```

规则：

```text
常用轻面板：KeepAlive = true
重面板：LazyLoad = true
Profiler / ShaderGraph / AnimationTimeline：首次打开再创建
关闭文档：释放 ViewModel、订阅、缓存、Bitmap
```

## 18.12 UI 线程策略

Avalonia 使用单 UI 线程模型，控件创建、布局、渲染和输入处理都发生在 UI 线程；后台线程访问控件会触发线程错误。官方文档建议从后台线程更新 UI 时使用 `Dispatcher.UIThread.Post` 或 `InvokeAsync`。([Avalonia 文档][5])

不要这样：

```csharp
foreach (var entity in entities)
{
    Dispatcher.UIThread.Post(() =>
    {
        VisibleRows.Add(new EntityRowViewModel(entity));
    });
}
```

这会把几万次任务塞进 UI 队列。

推荐使用 UI 批处理器：

```csharp
using System.Collections.Concurrent;
using Avalonia.Threading;

namespace Editor.UI.Performance;

public sealed class UiBatcher
{
    private readonly ConcurrentQueue<Action> _queue = new();
    private int _posted;

    public void Enqueue(Action action)
    {
        _queue.Enqueue(action);

        if (Interlocked.Exchange(ref _posted, 1) == 0)
        {
            Dispatcher.UIThread.Post(Flush, DispatcherPriority.Background);
        }
    }

    private void Flush()
    {
        try
        {
            while (_queue.TryDequeue(out var action))
            {
                action();
            }
        }
        finally
        {
            Interlocked.Exchange(ref _posted, 0);

            if (!_queue.IsEmpty &&
                Interlocked.Exchange(ref _posted, 1) == 0)
            {
                Dispatcher.UIThread.Post(Flush, DispatcherPriority.Background);
            }
        }
    }
}
```

后台线程做重活：

```csharp
var rows = await Task.Run(() =>
{
    return sceneIndex.BuildVisibleRows();
});
```

UI 线程只做一次批量替换：

```csharp
_uiBatcher.Enqueue(() =>
{
    VisibleRows.ReplaceRange(
        rows.Select(x => new EntityRowViewModel(x)));
});
```

## 18.13 高频事件必须节流

这些事件不能直接一变就刷 UI：

```text
Scene selection changed
Transform changed
Asset import progress
Console log appended
Profiler frame event
Mouse move
Drag over
Search result streaming
File watcher changed
```

建议统一做 `Throttle / Debounce / Batch`。

示例：Console 输出批处理。

```csharp
public sealed class ConsoleBuffer
{
    private readonly List<ConsoleMessage> _pending = [];
    private readonly RangeObservableCollection<ConsoleMessage> _visibleMessages;

    public ConsoleBuffer(RangeObservableCollection<ConsoleMessage> visibleMessages)
    {
        _visibleMessages = visibleMessages;
    }

    public void Append(ConsoleMessage message)
    {
        lock (_pending)
        {
            _pending.Add(message);
        }
    }

    public void Flush()
    {
        List<ConsoleMessage> messages;

        lock (_pending)
        {
            if (_pending.Count == 0)
            {
                return;
            }

            messages = [.. _pending];
            _pending.Clear();
        }

        _visibleMessages.AddRange(messages);
    }
}
```

然后每 50ms 或 100ms flush 一次，而不是每条日志刷新一次 UI。

## 18.14 内存和 GC 策略

游戏编辑器会产生大量临时对象，比如：

```text
搜索结果
日志消息
导入任务
缩略图 Bitmap
属性描述
拖拽数据
场景快照
Undo / Redo 数据
Profiler 采样
```

.NET 官方 GC 性能文档说明，托管堆上的对象分配率越高，GC 越频繁；降低分配率可以减少 GC 发生频率。([Microsoft Learn][6])

建议：

```text
1. 避免每帧创建字符串
2. 避免每帧创建 LINQ 临时集合
3. 避免每帧创建 Command / Binding / ViewModel
4. 避免大型 byte[] / float[] 反复创建销毁
5. 大数组优先复用或池化
6. Bitmap、Stream、文件句柄必须及时 Dispose
7. Undo / Redo 使用增量 diff，不要每次保存完整场景
```

特别注意 LOH。Microsoft 文档说明，大小大于等于 85,000 字节的对象会进入大对象堆；大型临时对象会带来分配和 Gen2/LOH 回收成本。([Microsoft Learn][7])

容易进入 LOH 的对象：

```text
byte[]
float[]
Vector3[]
大型 string
大型 JSON 文本
大型 Bitmap buffer
大型 undo snapshot
```

对于大数组，建议：

```text
小对象：正常分配即可
高频大对象：ArrayPool<T>
昂贵可复用对象：ObjectPool<T>
长期缓存对象：带容量上限的缓存
```

但对象池不要滥用。Microsoft 的 ObjectPool 文档明确提醒，对象池不总是提升性能；如果对象初始化成本不高，从池里取对象可能更慢，并且应该在真实场景测量后再使用。([Microsoft Learn][8])

## 18.15 Scene View 渲染策略

如果你要做游戏引擎编辑器，中间通常会有一个 Scene View / Game View。

不要用这种方式长期做主渲染：

```text
引擎后台渲染一张图片
每帧转换 Bitmap
每帧设置 Image.Source
让 Avalonia 每帧走普通图片控件刷新
```

更合适的是把视口做成专门的渲染控件：

```text
Features/SceneView
├── Views
│   └── SceneView.axaml
├── Controls
│   └── EngineViewportControl.cs
├── Services
│   └── SceneRenderBridge.cs
└── ViewModels
    └── SceneViewModel.cs
```

Avalonia 提供 `OpenGlControlBase`，它是用于 OpenGL 渲染控件的基类，负责 OpenGL 上下文管理、surface 创建和渲染生命周期。([Avalonia 文档][9])
如果只是做 2D 自定义绘制，Avalonia 自定义渲染文档说明可以重写控件的 `Render` 方法，并在数据变化时调用 `InvalidateVisual()` 请求重绘。([Avalonia 文档][10])

建议分层：

```text
SceneViewModel
    只保存相机状态、选中状态、工具状态

EngineViewportControl
    只负责承载渲染结果和输入

SceneRenderBridge
    负责 Avalonia UI 与引擎渲染线程之间通信

Engine Runtime
    真正执行渲染
```

注意：Scene View 的鼠标、键盘、拖拽输入可以进入 Avalonia；真正的渲染循环不要绑死在 Avalonia 的普通控件树更新上。

## 18.16 Binding 错误必须零容忍

Avalonia 性能文档明确指出，绑定错误会造成性能下降，因为每次绑定失败都会尝试解析并写入 trace log；文档还建议避免在 `DataTemplate` 里大量使用 `RelativeSource.FindAncestor`，可以用附加属性和属性继承替代视觉树查找。([Avalonia 文档][4])

因此建议设置规范：

```text
1. Debug 启动后不允许有 binding error
2. 大量 item template 必须使用 compiled binding
3. DataTemplate 必须写 x:DataType
4. 避免深层 RelativeSource.FindAncestor
5. 不在 item template 里绑定全局服务
6. 不在 item template 里创建复杂 converter
```

## 18.17 Converter 使用规范

大量实例下，Converter 也会成为热点。

不要这样：

```xml
<TextBlock Text="{Binding Value, Converter={StaticResource HeavyConverter}}" />
```

如果这个绑定出现在 10 万行里，converter 可能被调用很多次。

推荐：

```text
1. 简单格式化放 ViewModel
2. 高频转换预计算
3. Converter 必须无状态
4. Converter 不访问服务
5. Converter 不做 IO
6. Converter 不分配大量临时对象
```

例如：

```csharp
public sealed class EntityRowViewModel
{
    public string DisplayName { get; }
    public string IconKey { get; }
    public bool IsPrefabInstance { get; }

    public EntityRowViewModel(EntityRow row)
    {
        DisplayName = row.Name;
        IconKey = row.Kind switch
        {
            EntityKind.Camera => "camera",
            EntityKind.Light => "light",
            _ => "entity"
        };

        IsPrefabInstance = row.IsPrefabInstance;
    }
}
```

## 18.18 命令对象不要按行创建

不要在每个实体行里创建一堆命令：

```csharp
public ICommand RenameCommand { get; }
public ICommand DeleteCommand { get; }
public ICommand DuplicateCommand { get; }
public ICommand FocusCommand { get; }
```

如果 10 万行，每行 4 个命令，就是 40 万个命令对象。

推荐：

```text
行 VM 只保存 EntityId
命令放在父 ViewModel 或 CommandService
CommandParameter 传 EntityId
```

XAML：

```xml
<Button Command="{Binding $parent[UserControl].DataContext.DeleteEntityCommand}"
        CommandParameter="{Binding EntityId}" />
```

或者用命令服务：

```csharp
public Task DeleteEntityAsync(int entityId)
{
    return _commandService.ExecuteAsync(
        "scene.deleteEntity",
        new CommandContext(entityId));
}
```

## 18.19 事件订阅必须可释放

大量实例下，内存泄漏常常来自事件订阅。

危险写法：

```csharp
engine.SelectionChanged += OnSelectionChanged;
```

如果 ViewModel 关闭后没有取消订阅，它可能永远不会释放。

推荐：

```csharp
public sealed class SceneHierarchyViewModel : ViewModelBase, IDisposable
{
    private readonly ISceneSelectionService _selection;

    public SceneHierarchyViewModel(ISceneSelectionService selection)
    {
        _selection = selection;
        _selection.SelectionChanged += OnSelectionChanged;
    }

    public void Dispose()
    {
        _selection.SelectionChanged -= OnSelectionChanged;
    }

    private void OnSelectionChanged(object? sender, SelectionChangedEventArgs e)
    {
    }
}
```

Dock 文档面板关闭时，一定要释放：

```text
1. 事件订阅
2. CancellationTokenSource
3. Bitmap
4. 文件流
5. 大型缓存
6. 后台任务
7. 引擎对象句柄
```

## 18.20 后台任务规范

后台任务适合做：

```text
资源扫描
资源导入
缩略图生成
搜索
场景索引重建
脚本编译
日志解析
Profiler 数据聚合
```

Avalonia threading 文档建议重计算放到后台线程，完成后再回到 UI 线程更新界面；文档也提醒不要用 `.Result` 阻塞 UI 线程，应该使用 `async/await`。([Avalonia 文档][5])

推荐服务：

```csharp
public interface IBackgroundTaskService
{
    Task RunAsync(
        string name,
        Func<CancellationToken, Task> task,
        CancellationToken cancellationToken = default);
}
```

资源扫描示例：

```csharp
public async Task RefreshAssetsAsync()
{
    var assets = await Task.Run(() =>
    {
        return _assetIndexer.ScanWorkspace();
    });

    AssetItems.ReplaceRange(
        assets.Select(x => new AssetItemViewModel(x)));
}
```

不要在后台线程直接操作控件，也不要在后台线程直接改绑定集合。

## 18.23 性能检查清单

每个 Feature 合入前检查：

```text
1. 是否启用了 x:CompileBindings？
2. DataTemplate 是否写了 x:DataType？
3. 大集合是否启用了虚拟化？
4. ListBox / TreeView 是否被放进 StackPanel 导致虚拟化失效？
5. 是否有 binding error？
6. 是否逐条 Add 大量 ObservableCollection？
7. 是否在 UI 线程做 IO / 扫描 / 解析？
8. 是否对每个实例创建多个 Command？
9. 是否对每个实例订阅事件但不释放？
10. 是否有 Bitmap / Stream / CancellationTokenSource 没 Dispose？
11. 是否每帧创建字符串、数组、LINQ 临时集合？
12. 是否有大对象频繁进入 LOH？
13. 是否能用 dotnet-counters 看到 GC、CPU、异常率？
14. 面板关闭后内存是否能回落？
15. 大场景滚动是否稳定？
```

## 18.24 针对你的架构补充一句总原则

你前面的项目架构可以保留：

```text
Core
Infrastructure
UI
Shell
Features
```

但游戏引擎编辑器要额外加一条性能边界：

```text
Engine Runtime / Scene Data
    不直接暴露给 Avalonia UI

Feature Service / Index
    把引擎数据转换成可查询、可分页、可 diff 的结构

ViewModel
    只保存当前 UI 需要显示的轻量状态

View
    只创建当前可见控件
```

最终结构应该是：

```text
大量实例存在 Engine/Core
大量查询走 Index/Service
大量变化走 ChangeSet/Batcher
大量显示走 Virtualization
大量后台工作走 Task/Worker
少量可见状态进入 ViewModel
极少量控件进入 Visual Tree
```

这条原则比具体用什么控件更重要。游戏引擎编辑器能不能撑住大项目，关键就在于：**不要把引擎世界完整搬进 UI 树。**
