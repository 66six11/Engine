# Studio 项目规范

本文是 `apps/studio` 的执行规范。它描述当前 Avalonia 编辑器前端应该如何组织代码、拆分职责、维护 XAML/MVVM 边界和推进功能。`docs/编辑器UI组件.md` 是长期能力清单，本文才是当前阶段新增或重构代码时必须遵守的规范。

## 1. 规范定位

本文只描述工程边界、代码组织、性能约束和合入要求，不记录“当前已经实现到哪里”。包版本、已安装库、默认布局、现有类名、阶段进度和验证结果必须以源码、项目文件、lock 文件或对应专题指南为准，不同步进本文。

写文档时遵守：

```text
1. `docs/项目规范.md` 写长期规则、边界和准入条件。
2. 专题指南写实现事实、阶段路线和当前切片状态，例如 Dock 事实写入 `docs/Dock系统指南.md`。
3. 版本号只在需要锁定依赖或复现实验时出现在对应实现文档中。
4. 已存在的模板遗留问题可以在迁移任务或专题指南中记录，不把本文变成状态清单。
5. 新增代码仍必须按本文的目录、命名空间、MVVM、性能和验证规则执行。
```

## 2. 文档关系

`docs/architecture/README.md`：
Studio 正式架构文档入口。目标分层、生命周期、Edit/Play/Preview World、Viewport 和扩展模型以该目录为准。

`docs/adr/`：
记录同进程 EngineHost、跨平台 Viewport presentation 和项目边界等长期决策。修改相应方向前必须新增或替代 ADR。

`docs/编辑器UI组件.md`：
长期 UI 能力目录，包含 P0/P1/P2 能力清单。它不是第一阶段一次性实现清单。

`docs/architecture/studio-code-framework.md`：
当前 project/namespace、目标边界和迁移映射。代码放置不得依赖已删除的单项目分类快照。

`docs/项目规范.md`：
当前工程规范。写代码、拆目录、放 ViewModel、做控件、引入 Dock 或新增 Feature 时以本文为准。

`docs/编辑器UI平台规范.md`：
Partial 的 UI 平台实现规范。其 Dialog、Command、Shortcut 等局部规则继续有效，native viewport/Play/长期分层以正式架构文档为准。

如果能力清单和本文冲突，先按本文执行；如果功能目标已经变化，再同步更新两份文档。

## 3. 当前阶段目标

第一阶段只追求可运行、可扩展、边界清楚的编辑器外壳：

```text
1. 保持 Avalonia 应用可编译、可启动。
2. 收敛 Shell / UI / Core / Infrastructure / Features 的目录和命名空间。
3. 建立 View / ViewModel / 通用控件的放置规则。
4. 保留 Docking、Command、Panel、Feature 注册的扩展位置，并把第三方 UI 框架适配限制在 Shell 或 Infrastructure 边界内。
5. 为后续游戏引擎编辑器 P0 能力打基础，不提前实现完整编辑器。
```

禁止把 `docs/编辑器UI组件.md` 中的 52 项能力直接作为当前 Sprint 范围。每次只做一个可验证切片。

## 4. 分层目录

单项目阶段保留一个 `.csproj`，通过目录表达架构边界：

```text
Editor
├── App.axaml
├── App.axaml.cs
├── Program.cs
├── ViewLocator.cs
├── Assets
├── Core
├── Infrastructure
├── UI
├── Shell
├── Features
└── docs
```

各层职责如下：

```text
Core
  放跨 Feature 共享的抽象、核心模型、命令描述、事件契约。
  不引用 Avalonia 控件，不访问文件系统，不依赖 Shell、UI、Infrastructure 或 Features。

Infrastructure
  放文件系统、设置、持久化、序列化、日志、平台服务实现。
  实现 Core 中定义的接口，不承载具体 Feature 的业务 UI。

UI
  放通用视觉能力：基础控件、通用组件、样式、主题、转换器、行为。
  不能依赖具体 Feature。控件如果只服务单个 Feature，应放在该 Feature 内。

Shell
  放应用外壳：主窗口、全局布局、菜单、工具栏、状态栏、快捷键入口、全局命令入口。
  Shell 只编排能力，不写资源浏览器、Inspector、Scene View 等 Feature 内部逻辑。

Features
  放垂直功能切片。每个 Feature 自带 Views、ViewModels、Models、Services、Commands、Styles。
  Feature 之间不直接互相依赖，需要通信时通过 Core 抽象或 Shell 注册机制。
```

## 5. 依赖方向

允许的方向：

```text
App             -> Shell, Infrastructure, UI, Core, Features
Shell           -> Core, UI
Features        -> Core, UI
Infrastructure  -> Core
UI              -> Core
Core            -> 无业务层依赖
```

禁止的方向：

```text
Core             -> Avalonia / Shell / UI / Infrastructure / Features
Infrastructure   -> Shell / Features
UI               -> 具体 Feature
Feature A        -> Feature B
ViewModel        -> 直接 new View
Shell            -> Feature 内部业务实现
```

如果某个类型被两个以上 Feature 共享，才考虑上移到 `Core`、`UI` 或 `Infrastructure`。不要为了“以后可能复用”提前上移。

## 6. 命名空间规范

新增或重构文件时，命名空间必须跟逻辑目录一致：

```text
Shell/Views/MainWindow.axaml.cs
  namespace Editor.Shell.Views
  x:Class="Editor.Shell.Views.MainWindow"

Shell/ViewModels/MainWindowViewModel.cs
  namespace Editor.Shell.ViewModels

UI/Controls/Base/IconButton.axaml.cs
  namespace Editor.UI.Controls.Base
  x:Class="Editor.UI.Controls.Base.IconButton"

Features/AssetBrowser/ViewModels/AssetBrowserViewModel.cs
  namespace Editor.Features.AssetBrowser.ViewModels
```

现有模板命名空间可以分步迁移，但新增代码不得再放到错误命名空间下。迁移 `Shell` 命名空间时，要同步更新：

```text
App.axaml.cs
ViewLocator.cs
Shell/Views/MainWindow.axaml
Shell/Views/MainWindow.axaml.cs
Shell/ViewModels/*.cs
```

控件、ViewModel、服务不要使用容易和 BCL 类型冲突的名称。比如属性编辑器不要命名为 `String`，应使用 `StringPropertyEditor` 或 `StringPropertyEditorView`。

## 7. MVVM 与 ViewLocator

本项目使用 CommunityToolkit.Mvvm。默认规则：

```text
1. ViewModel 继承 ViewModelBase。
2. ViewModel 暴露状态、命令和轻量 UI 模型。
3. View 只负责显示、布局、控件交互和必要的 view-only 行为。
4. code-behind 只能写 InitializeComponent、控件内部逻辑、平台事件桥接或纯视图行为。
5. ViewModel 不直接引用 Avalonia 控件，也不创建 View。
```

当前 `ViewLocator` 使用反射按命名空间替换查找 View，这适合模板阶段，但不适合长期维护、裁剪和 AOT。后续新增 Feature 时优先采用显式映射：

```csharp
public Control? Build(object? data)
{
    return data switch
    {
        MainWindowViewModel => new MainWindow(),
        _ => new TextBlock { Text = $"Not Found: {data?.GetType().FullName}" }
    };
}
```

如果继续使用约定式查找，必须保证 View 和 ViewModel 命名空间可以稳定互相转换，并在重构目录时同步测试。

## 8. XAML 与绑定规范

项目已启用 Avalonia 编译绑定，所有新增 View 和 DataTemplate 必须遵守：

```text
1. View 根节点写 `x:DataType`。
2. DataTemplate 写明确的数据类型。
3. 需要动态绑定时显式标注并说明原因。
4. Debug 启动后不接受新增 binding error。
5. 不在大列表 item template 中使用昂贵 converter、深层 RelativeSource 查找或服务访问。
```

示例：

```xml
<Window xmlns="https://github.com/avaloniaui"
        xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
        xmlns:vm="using:Editor.Shell.ViewModels"
        x:Class="Editor.Shell.Views.MainWindow"
        x:DataType="vm:MainWindowViewModel">
    <TextBlock Text="{Binding Greeting}" />
</Window>
```

绑定失败要当作质量问题处理，不要等到性能优化阶段再修。

## 9. UI 控件规范

通用控件放在 `UI`，Feature 专属控件放在 Feature 内：

```text
UI/Controls/Base
UI/Controls/Inputs
UI/Controls/Properties
UI/Components
UI/Styles

Features/SceneView/Controls
Features/Inspector/Controls
Features/AssetBrowser/Components
```

控件类型选择：

```text
UserControl
  用于应用内组合型视图或 Feature 专属控件。

TemplatedControl
  用于可复用、可换模板、需要长期维护的通用控件。

Control + Render
  用于自定义绘制控件，例如曲线、网格、Gizmo overlay、轻量图标绘制。
```

通用控件不要只放占位文本。新增控件至少要定义用途、公开属性、样式入口和最小可运行示例。

## 10. Shell 规范

Shell 是编辑器外壳，不是业务功能集合。Shell 可以拥有：

```text
MainWindow
全局布局容器
主菜单入口
顶部工具栏入口
底部状态栏入口
快捷键入口
全局命令入口
面板注册入口
```

Shell 不应该拥有：

```text
文件树加载逻辑
资产导入逻辑
Inspector 属性生成逻辑
Scene View 渲染逻辑
Console 日志生产逻辑
搜索算法
构建流程内部逻辑
```

这些逻辑应进入对应 Feature 或 Infrastructure，再通过抽象暴露给 Shell 编排。

## 11. Docking 策略

Docking 是编辑器外壳能力，不是 Feature 业务逻辑。规范如下：

```text
1. Core 只定义面板描述、区域、类型、注册接口等与具体 Dock 框架无关的契约。
2. Shell/Docking 负责把面板注册表翻译为具体 Dock 布局，并隔离第三方 Dock 类型。
3. Feature 只注册自己提供的面板描述和轻量内容创建入口，不直接创建 Dock 控件。
4. 面板菜单路径保持 `Window/Panels/{PanelTitle}`，供后续命令系统和面板显示/隐藏使用。
5. Dock 主题定制放在 `UI/Styles/Shell/Dock*.axaml`，并映射到统一颜色 token，不在业务 View 中零散改色。
6. Dock 布局持久化只保存 ID、区域、比例、激活状态等轻量数据，不保存 ViewModel 或重对象。
7. 默认启动布局只创建当前验证 Dock 所需的少量轻量占位 ViewModel，不提前实例化真实重面板。
```

面板注册方向：

```text
PanelDescriptor
  Id
  Title
  Kind
  DefaultArea
  MenuPath
  CachePolicy
  CreateContent

PanelRegistry
  Register
  GetAll
  GetRequired
```

不要在 `MainWindowViewModel` 构造函数里一次性创建所有未来面板。启动路径只创建 Shell、主题、默认轻量状态和必要服务。Dock 的当前实现事实、库版本、默认布局和下一步计划写入 `docs/Dock系统指南.md`，不要同步回本文。

## 12. Feature 开发规范

新增 Feature 时使用垂直切片结构：

```text
Features/AssetBrowser
├── Views
├── ViewModels
├── Models
├── Services
├── Commands
├── Controls
└── Styles
```

Feature 必须满足：

```text
1. 对外只暴露必要的模块入口或服务接口。
2. 内部 ViewModel 不直接依赖其他 Feature。
3. 可复用契约放 Core，可复用视觉控件放 UI，可复用持久化实现放 Infrastructure。
4. 大量数据通过服务、索引、分页、虚拟化和批量更新进入 UI。
5. Feature 关闭或卸载时释放订阅、任务、Bitmap、Stream 和缓存。
```

第一阶段优先 P0 基础切片，例如 Shell、基础控件、状态栏、命令入口、简单资源浏览入口。P1/P2 能力必须等 P0 扩展点稳定后再进入。

## 13. 命令与输入规范

命令应成为菜单、工具栏、快捷键、右键菜单和命令面板的共同入口。当前阶段可以先保持轻量，但方向必须一致：

```text
CommandId
CommandDescriptor
ICommandService
ShortcutDescriptor
CommandContext
```

规则：

```text
1. UI 不直接调用复杂业务方法，优先绑定命令。
2. 高频列表行不要为每一行创建多组 ICommand。
3. 行项目只保存轻量 ID，命令放在父 ViewModel 或命令服务。
4. 异步命令使用 AsyncRelayCommand 或等价封装，不能阻塞 UI 线程。
5. Undo / Redo 后续应接入同一命令体系，不另起一套入口。
```

## 14. 性能规范

游戏引擎编辑器不能把引擎世界完整搬进 Avalonia 控件树。基本原则：

```text
大量实例存在 Engine/Core
大量查询进入 Index/Service
大量变化进入 ChangeSet/Batcher
大量显示依赖 Virtualization
大量后台工作进入 Task/Worker
少量可见状态进入 ViewModel
极少量控件进入 Visual Tree
```

必须遵守：

```text
1. 大集合使用虚拟化，不能放进给无限高度的 StackPanel。
2. 不逐条 Add 几千个 ObservableCollection 项来刷新 UI。
3. 搜索、扫描、导入、缩略图生成、解析和排序不在 UI 线程执行。
4. 高频事件必须 throttle、debounce 或 batch。
5. Bitmap、Stream、CancellationTokenSource 和事件订阅必须可释放。
6. Console、Problems、Search Results 等面板必须有容量上限或分页策略。
7. Inspector 动态属性 UI 只为当前展开、可见、正在编辑的内容创建控件。
8. Scene View 渲染循环不绑定到普通 Avalonia Image 每帧换图。
```

后台线程完成计算后，只把最终 UI 更新切回 UI 线程。

## 15. 资源与持久化规范

持久化模型和业务模型分开：

```text
Core/Models
  Workspace
  DocumentId
  EditorDocument
  PanelDescriptor

Infrastructure/*/Models
  SettingsFileModel
  WorkspaceFileModel
  RecentFileEntryDto
  DockLayoutSnapshot
```

`Core` 描述编辑器概念，`Infrastructure` 描述“怎么存”。不要把 JSON 结构、文件格式 DTO、平台路径细节塞进 `Core`。

## 16. 编码与格式

遵守仓库编码约束：

```text
Markdown、XAML、C#、csproj、json、ps1 等非 C/C++ 文件使用 UTF-8 without BOM。
C/C++ 源文件规则以仓库根 AGENTS.md 和 tools/check-text-encoding.ps1 为准。
```

Markdown 文档使用中文说明，代码标识、命令和路径保持原文。不要提交乱码文档，不要提交 IDE 自动大范围重排。

## 17. 验证要求

文档改动至少检查：

```powershell
git diff --check
```

C# / XAML 改动至少检查：

```powershell
dotnet build
```

如果新增测试项目或已有测试项目可用，再运行：

```powershell
dotnet test
```

UI 敏感改动还需要启动应用，检查：

```text
1. 窗口能打开。
2. Debug 输出没有新增 binding error。
3. 新增控件在默认主题下可见、可交互、无明显布局溢出。
4. 大集合面板滚动时没有明显卡顿或控件数量失控。
```

## 18. 合入前检查清单

每次合入前检查：

```text
1. 文件是否放在正确目录。
2. namespace / x:Class 是否匹配目录。
3. View 是否写了 x:DataType。
4. ViewModel 是否避免直接引用 View 或 Avalonia 控件。
5. 通用代码是否真的需要上移到 Core / UI / Infrastructure。
6. Feature 是否直接依赖了另一个 Feature。
7. 大集合是否虚拟化、分页或批量更新。
8. 后台任务是否避免阻塞 UI 线程。
9. 订阅、任务、Bitmap、Stream 是否可释放。
10. 文档是否同步说明了新增边界、依赖或阶段范围。
```

不确定放在哪里时，先放在最具体的 Feature 内。等第二个真实使用场景出现，再上移。
