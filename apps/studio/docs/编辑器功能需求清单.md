所有会改变项目数据的功能，都应明确：

- 是否进入 Undo/Redo。
- 是否影响 Dirty State。
- 如何保存、加载、恢复。
- 失败时如何提示和回滚。
- 是否需要验证、日志或诊断证据。

## 编辑器产品原则

- 编辑器主循环应覆盖 `查找 -> 检查 -> 选择 -> 编辑 -> 预览 -> 验证 -> 提交/撤销`。
- 编辑器状态与运行时数据分离：布局、选择、过滤器、面板状态不应污染可发布资产。
- 持久化变更必须经过命令、事务或等价机制，保证 Undo/Redo、Dirty State 和保存语义一致。
- 面板与工具通过稳定贡献点接入，避免核心 Shell 直接依赖具体功能模块。
- 需求验收以用户可完成的工作流为准，不只以类、接口或控件存在为准。

## 扩展边界草案

当前阶段先把内部受信任扩展跑通，不承诺第三方插件 ABI/API、外部程序集热加载、任意 XAML 安全沙箱或脚本 VM。

长期扩展形态保留两条 UI 描述路径：

- XAML UI：用 XAML 或等价声明式格式描述面板、Inspector、工具窗口和属性编辑布局，由 Shell 负责加载、主题资源、绑定上下文、生命周期、错误诊断和失败回滚。
- Code-first UI：用纯代码描述编辑器 UI，类似 Unity 的代码式 EditorWindow/Inspector 思路。扩展可以注册命令、打开面板，并在面板中声明文本、输入框、按钮、列表、属性行、资源选择器等基础控件。

两种方式都必须落到统一贡献模型：

```text
XAML UI / Code-first UI
    -> contribution descriptor
    -> validation
    -> Shell registry
    -> panel / command / inspector / tool lifecycle
```

Code-first UI 不是绕过 Shell 直接持有任意窗口或全局状态。它应通过受控 UI builder、ViewModel 或声明式对象模型描述界面；持久化修改仍然通过命令、事务、验证、Dirty State 和 Undo/Redo 接入。

## 后端底层依赖清单（渲染调试 MVP）

本节只记录编辑器渲染调试 MVP 依赖的后端能力，不把后端实现并入编辑器 Slice。编辑器需求通过 `Dependencies` 引用这些编号；后端能力应单独拆成 Runtime/Renderer/RHI/Asset Pipeline 侧的 `[Slice]`。

| ID | Backend Capability | Layer / Owner | Editor Uses | Contract / Acceptance | Priority | Stage | Status | Notes |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| BE-RDBG-001 | 帧统计 API。 | Renderer / RHI diagnostics | Viewport 状态栏、渲染调试面板、性能概览。 | 每帧可读取 FPS、CPU frame time、GPU frame time、frame index、frames-in-flight；缺失 GPU timer 时明确返回 unavailable。 |  | MVP |  | 不要求首版 profiler，只要求稳定摘要。 |
| BE-RDBG-002 | 渲染路径标识。 | Renderer frontend | 区分 Render Basic、未来 SRP、debug/fallback 路径。 | 每个 Viewport/Frame 可查询 renderer path、backend name、active render mode、feature flags；编辑器可用它决定显示哪些调试入口。 |  | MVP |  | Render Basic 保留期间必须能被明确标识为过渡路径。 |
| BE-RDBG-003 | RenderGraph pass 列表。 | RenderGraph / Renderer frontend | Frame Debugger pass 列表、pass 选择、pass 状态展示。 | 每帧可枚举 pass id、name、enabled/skipped 状态、输入输出资源、队列/阶段摘要；pass id 在一帧内稳定。 |  | MVP |  | 不要求暴露 Vulkan barrier/layout 细节到编辑器。 |
| BE-RDBG-004 | 渲染资源列表。 | RenderGraph / Resource manager | 资源/纹理检查器、pass 输入输出跳转。 | 可按 frame/pass 查询资源 id、name、type、format、extent、usage、producer/consumer；资源失效时可诊断。 |  | MVP |  | 不暴露裸后端 handle，使用稳定 debug id 或抽象引用。 |
| BE-RDBG-005 | pass 结果预览。 | Renderer debug preview | 点击 Frame Debugger pass 后，在 Viewport 或调试窗口显示该 pass 输出。 | 选中 pass 输出时可提供可显示的 debug texture/view；未选中 pass 时 Viewport 继续正常渲染并保持 timer 流动。 |  | MVP |  | 这是 Q-003 的后端依赖，不应由编辑器伪造截图。 |
| BE-RDBG-006 | shader/material 元数据。 | Shader pipeline / Material system | Shader、Material、Pipeline 兼容性检查和调试显示。 | 可查询 shader stage、entry point、source/artifact id、reflection binding、material parameter、compile status/error。 |  | Alpha |  | MVP 可先只显示已存在元数据，完整材质编辑可后置。 |
| BE-RDBG-007 | 错误/日志回传。 | Renderer / RHI / Shader pipeline diagnostics | Console、Problems、渲染调试错误定位。 | 后端诊断包含 severity、source、frame/pass/resource/material/shader context、human-readable message；编辑器可过滤并跳转。 |  | MVP |  | Vulkan/backend 错误需保留上下文，但不把 API 细节变成 UI 主模型。 |
| BE-RDBG-008 | 调试快照生命周期。 | Renderer diagnostics | 保证 frame/pass/resource 调试数据可被 UI 安全读取。 | 编辑器读取快照不会阻塞 render loop；快照有明确生命周期、过期状态和失败提示。 |  | MVP |  | 避免为调试 UI 引入隐藏 GPU stall。 |

Issue 拆分建议：

- `BE-RDBG-*` 使用后端 Slice 承接，例如 `[Slice] Renderer diagnostics: frame stats API`。
- `ED-VIEW-*` / `ED-DIAG-*` 使用编辑器 Slice 承接，例如 `[Slice] Studio frame debugger pass list panel`。
- 编辑器 Slice 对后端 Slice 使用 `Refs #N`，只有端到端验收完成时才使用 `Closes #N`。
- 后端 Slice 的 Done evidence 应包含 smoke、日志样例、调试数据样例或截图；编辑器 Slice 的 Done evidence 应包含 UI 操作路径和空数据/失败数据状态。
